# v0.3.13-alpha

Study Anything v0.3.13-alpha hardens the deployment path for external users,
platform Agents, and other LLM operators. The goal is to make the first useful
run prefer Skill Mode or published images, while keeping source builds as the
contributor path.

## What changed

- Adds `deployment-hardening-verification-v1`, a redacted machine-readable
  report that validates the launch scripts, diagnostics, clean-clone smoke,
  published-image fallback, Compose boundaries, platform packs, and adoption
  artifacts.
- Wires the deployment hardening verifier into release checks, ecosystem
  submission evidence, platform packs, adoption pack, and bundle manifest.
- Documents the operator path more explicitly: first try Skill Mode or
  `USE_PUBLISHED_IMAGES=true`, then use source builds only when changing the
  API image.
- Keeps the non-ASCII checkout guard and GHCR pull-timeout fallback visible as
  acceptance evidence instead of ambiguous local failures.

## Validation

Run the local release gate:

```bash
./scripts/release_check.sh
```

Deployment-specific checks:

```bash
python3 scripts/verify_deployment_hardening.py --check
python3 scripts/verify_deployment_hardening.py --pack platform/generated/study-anything-platform-adoption-pack.zip
docker manifest inspect ghcr.io/jzvcpe-goat/study-anything/api:v0.3.13-alpha
python3 scripts/verify_published_image_launch.py --tag v0.3.13-alpha --pull-timeout-seconds 180 --allow-pull-timeout-report
```

## Known limits

- A local GHCR pull timeout can still happen on slow Docker/GHCR networks. It is
  acceptable only when GitHub `docker-images` passed for the same tag, `docker
  manifest inspect` shows `linux/amd64` and `linux/arm64`, and release checks
  passed before tagging.
- Real model credentials remain outside Study Anything, inside the user-owned
  Agent or platform runtime.
- The standalone frontend remains out of scope for this release path.
