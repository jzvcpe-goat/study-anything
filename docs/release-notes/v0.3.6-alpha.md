# v0.3.6-alpha

Study Anything v0.3.6-alpha hardens the local recovery and backup safety layer.
The goal is to make self-host operators and platform Agents more confident that
backup evidence, restore previews, and recovery diagnostics are safe to share
without exposing local paths or private learning data.

## What Changed

- Added `security-recovery-hardening-verification-v1` in
  `scripts/verify_security_recovery_hardening.py`.
- Hardened backup manifest verification against:
  - unsafe relative paths such as `../escape`
  - absolute paths
  - backslash path tricks
  - duplicate file records
  - invalid sha256 values
  - missing or tampered backup files
- Redacted backup verification errors so diagnostics use manifest-relative names rather than absolute
  local backup paths.
- Kept `file_record(...)` constrained to files inside the backup directory.
- Verified encrypted sync wrong-passphrase and ciphertext-tamper errors stay redacted.
- Verified restore-preview remains count-only and keeps destructive restore outside the API.
- Wired the new verifier into release checks, external adoption proof, ecosystem submission, platform
  packs, operator drill evidence, and the adoption pack.

## Verify

```bash
python3 scripts/verify_security_recovery_hardening.py
python3 scripts/verify_backup_restore_drill.py
python3 scripts/verify_ecosystem_submission_pack.py
python3 scripts/verify_platform_ecosystem_packs.py
python3 scripts/generate_platform_bundle_manifest.py --check
python3 scripts/generate_platform_adoption_pack.py --check
./scripts/release_check.sh
```

Expected recovery proof:

- valid backup manifests verify successfully
- tampered files and invalid digests fail
- unsafe manifest paths are rejected before file access
- diagnostics do not include absolute temp/project paths
- restore-preview returns counts, hashes, and warnings only
- destructive restore remains manual and CLI-only

## Known Limits

- The verifier is offline and does not replace the existing Docker backup/restore drill.
- Local backups can still contain private data and `.env` secrets by design; operators must keep them
  encrypted at rest and out of public repos.
- Hosted recovery, account recovery, remote encrypted storage, and support workflows remain future
  hosted-service work.
